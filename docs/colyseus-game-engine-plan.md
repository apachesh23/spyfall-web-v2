# План: игровой движок на Colyseus + новые таблицы Supabase

Документ фиксирует архитектурные решения, соответствие текущей базе (`docs/current-db`) и поэтапное наращивание логики. Старые таблицы `game`, `game_events`, `voting_*`, `spy_guess_votes` **не трогаем** — параллельно вводим новый контур.

---

## Зафиксированные решения

- **Префикс новых игровых сущностей в БД — `match_*`** (например `match_sessions`, `match_events`). Не смешиваем с legacy `game*`.
- **MVP: автовосстановление матча после падения game-server не делаем.** Объём (снапшоты состояния, версии схемы, rehydrate комнаты, края с отвалившимися клиентами) для короткой партии не окупается.
- **Короткий матч (~15 минут):** допустимо зафиксировать продуктовое правило «упал сервер / деплой — партия потеряна». Вероятность обрыва не «каждую минуту», но **ненулевая**: рестарты, деплои, OOM, перенос инстанса в оркестраторе — всё это для Colyseus выглядит как конец комнаты в памяти.
- **Длинные сессии (условная монополия 1–2 часа):** там **без периодических снапшотов и/или сценария восстановления** уже рискованно с точки зрения ожиданий игроков; для Spyfall это ориентир «на будущее», не требование MVP.
- **Что в MVP из персистентности всё же полезно сразу:** строка **`match_sessions`** (старт/конец, комната, снапшот настроек) + **append-only `match_events`** (или эквивалент) для пост-игры и отладки — см. раздел 2 ниже. Таблицу **`match_state_snapshots`** в MVP можно **не** вводить, пока не понадобится recovery или глубокий аудит.

## 1. Новые таблицы вместо миграции старых — нормально?

**Да, это разумный выбор**, если цель — не ломать прод и наращивать движок миграциями только «вперёд».

| Плюсы | Минусы |
|--------|--------|
| Нет риска сломать существующий поток, если он ещё где-то живёт | Два контура в голове, пока не выключите старый путь полностью |
| Схема под Colyseus и серверные state machine — с нуля, без legacy-полей | Нужны явные префиксы/имена; **зафиксировано: `match_*`**, чтобы не пересечься с legacy и будущими именами |
| Можно добавлять таблицы маленькими миграциями по фичам | История/аналитика по старым играм остаётся в старых таблицах, пока не сделаете отдельный отчёт/ETL при необходимости |

Итог: **меньше багов от «полумигрированной» схемы**; больше дисциплины в именовании и в том, какой код пишет в какие таблицы.

## 2. «Все расчёты на сервере» — что с Supabase?

**Истина в матче — процесс Colyseus** (`SpyfallRoom` + `@colyseus/schema`). Клиент только отправляет намерения (голос, кнопка, текст угадывания), сервер валидирует и меняет состояние.

**Это не значит, что Supabase «не нужен».** Во время матча расчёты и состояние — в Colyseus; Supabase остаётся для **лобби**, **справочников**, **идентичности игроков/комнат** и **послеигровой статистики / аудита**, если вы туда пишете события.

### Что нужно от Supabase для MVP (практичный минимум)

| Область | Зачем |
|--------|--------|
| **Лобби** | `rooms`, `players`, `rooms.settings`, инвайты, статус «до старта» — без этого веб-часть комнаты не живёт |
| **Справочники** | `locations` (и аналоги) — чтение при подготовке матча |
| **`match_sessions`** | Одна строка на партию: `room_id`, `started_at` / `ended_at`, `settings_snapshot`, `rng_seed` (по желанию), итог (`winner`, `aborted`) — якорь для истории и UI «после игры» |
| **`match_players`** (опционально, но удобно) | Снимок состава на старт + итог: `player_id`, `was_spy`, финальное место/выбытие — чтобы не вытаскивать роли только из сотен событий |
| **`match_events`** | Append-only лог: что произошло в матче. Сюда же укладывается **пост-игровая статистика**: кто кем был (если не дублируете только в `match_players`), **за кого голосовал** (`vote_cast` с `voter_id`, `target_id`, фаза/раунд), изгнания, исход угадывания локации, подтверждения «да/нет» и т.д. — всё в `type` + `payload jsonb` |

Итого: для MVP **не «только ивенты» в вакууме**, а **лобби + справочники + одна сущность матча + поток событий** (и при желании нормализованная строка на игрока в матче). Если **вообще** не писать в БД ничего из игры — не будет ни истории партии, ни нормальной страницы «кто кого выгнал», кроме как держать всё в памяти до закрытия вкладки.

### Лобби и `match_*`: не путать с «дублированием ради Colyseus»

- **До старта матча** источник правды по комнате и списку людей — как и сейчас, **`rooms` / `players` в Supabase** (лобби не выкидываем и не заменяем таблицами `match_*`). Речь «не трогать» в контексте legacy относится к **старым `game*`-таблицам**, а не к лобби.
- **Во время матча** game-server **знает состав и состояние из памяти** `SpyfallRoom` + `@colyseus/schema` и подключённых клиентов. Ему **не нужно** постоянно читать `match_players` из Postgres, чтобы «понимать, кто играет».
- **`match_sessions` / `match_players` в БД** — это не второй лобби-слой, а **снимок на момент старта партии** + **итоги/удобство для пост-игры** (и опционально якорь `match_session_id` для `match_events`). Запись делается в момент **start match** (и при необходимости обновление итоговых полей в конце), а не как замена лобби-трафика в реальном времени.
- Как передать состав в Colyseus при старте: типично **один раз** — Next.js API / сервис читает лобби, создаёт `match_sessions` (+ строки `match_players` при желании), отдаёт клиенту `matchId` + опции join; **или** game-server один раз дергает Supabase по service role при `onCreate` — на выбор архитектуры, но **истина в рантайме матча всё равно в комнате**.

**Supabase после перехода на Colyseus (сводка ролей):**

| Роль | Зачем |
|------|--------|
| Лобби до старта | Уже есть: `rooms`, `players`, `settings`, справочники (`locations` и т.д.) — создание комнаты, инвайт, настройки |
| Снимок на старт матча | Один раз зафиксировать `settings` + список участников (id, nickname, avatar), чтобы сервер не зависел от гонок в лобби |
| Послеигровая аналитика и экран истории | `match_sessions` + `match_events` (+ опционально `match_players`); голоса и роли — из событий и/или итоговых полей |
| **Не автоматическая** «подстраховка при падении» | Если game-server упал, **комната в памяти исчезает**, пока вы сами не реализуете восстановление. Запись в БД сама по себе ничего не восстановит (в MVP это осознанно не целимся) |

### Восстановление сессии после падения — как это реально делается

1. **Периодические снапшоты** (например раз в N секунд или на каждый важный переход фазы): в строку матча пишется `state_snapshot jsonb` + `version`, `updated_at`.
2. **При старте матча** — создаёте запись `matches` (или аналог), сохраняете `colyseus_room_id` / `match_id`, который клиенты передают при join.
3. **После краша** — новый процесс поднимает **новую** комнату Colyseus; опциональный поток «rehydrate»: загрузить последний снапшот из БД и вызвать кастомную инициализацию состояния (это уже ручная логика, не из коробки).
4. **MVP без восстановления** — зафиксировано: «падение / деплой = матч потерян»; при этом **пост-игровая история** по завершении партии пишется в `match_*` (см. выше).

**Вывод:** **снапшоты состояния для rejoin** имеют смысл, когда явно проектируете восстановление (после MVP или для длинных игр). Для текущего Spyfall-MVP достаточно **лобби + `match_sessions` + `match_events`** (и опционально `match_players`).

## 3. Связь с текущей базой (`docs/current-db`)

Уже есть полезные кирпичи:

- **`rooms`**: `settings` (jsonb), `status`, `current_game_id` (legacy), `splash_event` (jsonb) — для **чистого Colyseus-потока** сплэши логичнее вести в **состоянии комнаты** и дублировать в БД только если нужен офлайн-просмотр/аналитика.
- **`players`**: `is_spy`, `is_alive`, `role`, `wants_early_vote` — это **клиентско-supabase эра**; в новом движке игровые флаги должны жить в **серверном state** и при необходимости **зеркалиться** в новые таблицы, а не быть единственным источником правды во время матча.
- **`locations`**: справочник локаций/тем/ролей — **остаётся в Supabase**; сервер при старте матча выбирает локацию и роли **детерминированно на сервере** (random seed в state + логирование seed в `matches` при желании fair audit).

## 4. Организация логики под Colyseus (модули в голове и в коде)

Рекомендуемая структура **внутри `SpyfallRoom`** (или подклассов/композиции):

1. **`MatchConfig`** — заморозка настроек из лобби (длительность таймера, число шпионов, режимы, включён ли kill vs guess и т.д.).
2. **`PhaseController` / state machine** — дискретные фазы: `lobby` (если комната объединяет лобби+игру) или только игровые: `playing` → подфазы голосования и т.д.
3. **`TimerService`** — один источник времени окончания раунда/матча на сервере.
4. **`VotingService`** — досрочное и финальное голосование, переголосование, tie-break.
5. **`SpyActionsService`** — угадывание локации (автопроверка + фаза подтверждения игроками), убийство (отдельный режим).
6. **`SplashService`** — очередь или одно активное событие сплэша с типом, полезной нагрузкой и дедлайном показа.

Клиент: подписка на `@colyseus/schema`, UI только отображает `phase`, `timer`, `activeSplash`, маскированные данные по игроку.

## 5. По вашим системам — краткая спецификация на сервере

### 5.1 Глобальный таймер

- Хранить **`matchEndsAt` (server timestamp ms)** или **`remainingMs` + last tick** (предпочтительно `endsAt` — проще при реконнекте и снапшотах).
- Обновлять время **только на сервере** (`setSimulationInterval` или таймер); раз в тик/интервал рассылать патчи.
- Синхронизация «100%» у всех: клиент показывает `endsAt - now` с поправкой **RTT/offset** (у вас уже есть задел: `ping` / `pong` с `serverTime` в `SpyfallRoom`).
- Значение длительности брать из **снимка настроек** при `match_start`, не из «живого» json в комнате лобби.

### 5.2 Роли, локация, темы, RP-роли, режимы из лобби

- При старте: сервер читает зафиксированный `MatchConfig`.
- Назначение шпионов/мирных — **только сервер**; в схему для клиента уходит то, что разрешено видеть (шпиону — не локация и т.д. по правилам).
- Рандом: **`rngSeed`** в состоянии матча + детерминированный порядок (удобно для отладки и повторяемости в тестах).

### 5.3 Досрочное голосование

**a) Кнопка early-vote**

- Считать порог от **живых игроков в дискуссии** (или от стартового состава — зафиксировать в правилах).
- Сервер ведёт множество `playerId → requestedEarlyVote`; при достижении **> 50%** — переход в фазу голосования (или в подфазу «подготовка»).
- Ограничения: **макс. 2 досрочных за матч**, **перезарядка 3 мин** между **успешными** триггерами (уточнить: кулдаун от последнего *запуска* голосования или от окончания — заложить в конфиг).
- **Первое досрочное не раньше 3 минут после старта** — отдельное поле `firstEarlyVoteNotBefore` от `matchStartedAt`.

**b) Голосование и исходы**

- Бюллетени только на сервере; анонимность по желанию — тоже серверное правило.
- Выгнали шпиона → **немедленный конец игры** (победа мирных).
- Выгнали мирного → если **минимальный состав для продолжения** выполнен — возврат в `playing` / discussion; иначе конец по правилам.
- **Переголосование при ничьей по двум лидерам** — отдельная подфаза `runoff` с узким списком кандидатов; проще всего **голосовать только между ними** (остальные варианты игнорировать в этой подфазе).

**c) Повторная ничья 50/50**

- Досрочное: **голосование не состоялось** → возврат в игру, расход лимита досрочных/кулдаун — зафиксировать в дизайне (обычно лимит тратится, если голосование было инициировано).

**d) Лимиты**

- В `MatchConfig`: `maxEarlyVotes`, `earlyVoteCooldownMs`, `firstEarlyVoteDelayMs`.

### 5.4 Финальное голосование

- Триггер: **`matchEndsAt` достигнут** (или фаза `finalVoting` с отдельным коротким таймером на голосование — по UX).
- Логика ветвлений как у досрочного, **но без исхода «не состоялось»**: при tie — снова `runoff`, пока не будет победителя по правилам (или жёсткий лимит итераций с запасным правилом — заложить в ТЗ, чтобы не зациклиться в теории).
- Любой исход **завершает матч** с победой команды.

### 5.5 SplashScreen

- Название нормальное; в коде удобно: `MatchBanner`, `Interstitial`, `SplashEvent`.
- В схеме: `activeSplash: { type, payload, dismissAt? }` или очередь `splashQueue[]` с серверным `shift` после таймаута.
- Критично: **один источник** — сервер решает «показать изгнание», клиенты только анимируют.

### 5.6 Шпион: угадывание локации

- Лимит **2 попытки за игру**, **кулдаун 3 мин** между попытками — счётчики и `nextGuessAt` на сервере.
- Текст → **серверный** fuzzy-match (ваш существующий модуль переносится/импортируется в `game-server`).
- Автозачёт / отказ → если серая зона — фаза **`spyGuessConfirmation`**: всем (или только мирным — по правилам) показать ввод, таймер **~60 с**, голоса ДА/НЕТ.
- Исход влияет на победу шпиона/продолжение — явно прописать в state machine.

### 5.7 Шпион: убийство (режим)

- **Один раз за игру**; **взаимоисключение** с «угадайкой» — флаг `spyConsumableAction: 'none' | 'guess' | 'kill'` или две блокировки, которые нельзя обойти с клиента.

## 6. Предлагаемые новые таблицы Supabase (черновик)

Имена можно сменить; важен **префикс** и однозначный FK на `rooms.id`.

| Таблица | Назначение |
|---------|------------|
| **`match_sessions`** | Один ряд на матч: `id`, `room_id`, `started_at`, `ended_at`, `status` (`active` / `finished` / `aborted`), `settings_snapshot jsonb`, `rng_seed`, опционально `colyseus_process_id` / `room_id_colyseus` |
| **`match_state_snapshots`** | `match_session_id`, `version`, `state jsonb`, `created_at` — восстановление после краша и глубокий аудит (**после MVP**, см. «Зафиксированные решения») |
| **`match_events`** | Append-only: `match_session_id`, `seq`, `type`, `payload jsonb`, `created_at` — аналитика, спорные моменты, будущий «replay» лобби-истории |
| **`match_players`** | Снимок состава на старт: `match_session_id`, `player_id` (uuid из `players`), `nickname`, `avatar_id`, итоговые `was_spy`, `placement` — удобно для пост-игры без парсинга огромного json |

**Голосования и бюллетени** в БД нужны, если хотите историю/модерацию; иначе на первом этапе достаточно **`match_events`** с типами `vote_cast`, `vote_phase_ended`. Нормализованные `voting_ballots_v2` имеют смысл, когда понадобятся отчёты SQL.

**Старая `game_history`** — аналог можно строить из `match_events` + `match_sessions` без касания legacy.

## 7. План действий по шагам (итерации)

1. **Контракт матча**: типы в `packages/shared` — фазы, сообщения клиент↔сервер, минимальная `schema` (игроки, таймер, фаза).
2. **Старт матча**: API или сообщение хоста → создание `match_sessions` + snapshot настроек из `rooms.settings` + список игроков.
3. **Таймер + фаза `playing`** без голосований — проверка синхронизации и реконнекта.
4. **Роли и раздача карточек** (маскирование на клиенте).
5. **Досрочное голосование** (порог 50%, лимиты, кулдауны).
6. **Переголосование и ничья** для досрочного.
7. **Финальное голосование** по таймеру.
8. **Сплэши** на ключевые переходы.
9. **Угадывание локации** + фаза подтверждения.
10. **Режим убийства** и взаимоисключение с угадайкой.
11. **Пост-игровой экран и статистика** — запись в `match_events` по ходу матча (голоса, изгнания, роли на конец и т.д.), чтение агрегатов; при необходимости денормализация в `match_players` на `match_ended`.
12. **По необходимости после MVP:** `match_state_snapshots` + сценарий восстановления после рестарта сервера (не блокирует первый релиз движка).

## 8. Риски и явные решения «закрыть в ТЗ»

- **Что считать «игроком» для 50%** — только живые? исключить отключившихся без паузы?
- **Кулдаун досрочного** — от какого момента тикает.
- **Финал при вечном tie** — верхняя граница раундов переголосования + запасное правило.
- **Отключение игрока** — пауза матча или бот/автовыбытие — отдельное правило.
