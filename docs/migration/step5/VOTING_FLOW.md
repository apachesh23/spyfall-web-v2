# Поток голосования (VotingSplash)

## Таймер "Осталось... N"
- Время, в течение которого игроки могут отдать голоса.
- Следующий этап запускается при **любом** из условий:
  - a) таймер дошёл до 0;
  - b) все игроки проголосовали (не ждём таймер).
- В текущей архитектуре переход решает **БД**:
  - `voting_ballots` trigger вызывает `resolve_voting_session(session_id)`,
  - либо `GET /api/game/vote/snapshot` дожимает просроченную collecting-сессию.

## После окончания периода голосования — 3 исхода

### 1) Голосование не состоялось
- **Условие:** никто не проголосовал или все нажали «Пропустить» (или повторное голосование снова в ничью).
- **Экран:** список и кнопки скрыты, в центре текст «Голосование не состоялось».
- **Низ:** «Игра продолжается...» и обратный отсчёт 5→0 с.
- Держим 5 с, затем закрываем сплэш (игра уже возобновлена на бэкенде).

### 2) Есть явный победитель
- **Условие:** один игрок набрал больше всего голосов.
- **Экран:** список и кнопки скрыты, в центре одна карточка победителя; вместо иконки кулака — процент (например 75%).
- **Низ:** зависит от исхода:
  - ранний раунд: промежуточный экран и затем продолжение/финал,
  - финальные ветки: переход в итоговый splash game over.

### 3) Ничья (первое голосование)
- **Условие:** у двух (или более) игроков поровну голосов.
- **Экран:** список скрыт, в центре только 2 карточки этих игроков.
- **Низ:** «Продолжение..» 5→0 с.
- После 5 с **не** закрываем сплэш, а переходим к повторному голосованию.

## 3.2) Повторное голосование
- Новый таймер 60 с «Осталось... N».
- В центре — все карточки игроков, но **выбирать можно только из двух кандидатов**; остальные полупрозрачные и без акцентного бордера при клике.
- Логика кнопок и кулаков та же.
- Исходы:
  - **3.2.1)** Снова ничья → считаем «Голосование не состоялось» → как пункт 1.
  - **3.2.2)** Один набрал больше → как пункт 2.

## Бэкенд
- Основа: `voting_sessions` + `voting_ballots`.
- Единая authoritative RPC: `resolve_voting_session(session_id)`.
- Внутри SQL:
  - проверка готовности к резолву,
  - claim `collecting -> processing`,
  - выбор ветки (`tie_revote` / `tie_failed` / `no_elimination` / `eliminated` / `final_civilians_lose`),
  - применение side-effects (`games`, `players`, `rooms`, `voting_sessions`).
- Transport voting: `postgres_changes` + snapshot fallback (`GET /api/game/vote/snapshot`).
- `voting_state` broadcast для voting больше не используется.

## Клиент (состояние)
- `votingPhase`: 'voting' | 'result_no_vote' | 'result_winner' | 'result_tie' | 'revote' | 'revote_result_no_vote' | 'revote_result_winner'
- `votingResult`: объект результата с сервера.
- `countdown5`: 5→0 для экранов с фразами «Игра продолжается» / «Продолжение».
- Клиент не инициирует `finishVoting()` по таймеру.
- При `votingStatus=active` идёт polling snapshot (~2.5с) + realtime из `postgres_changes`.

## Что тестить (чеклист)
- Все живые проголосовали до таймера: результат появляется сразу (без ожидания ручного ping).
- Время истекло, не все проголосовали: резолв по текущим бюллетеням.
- Ничья в 1 раунде: переход в revote с 2 кандидатами.
- Ничья в revote: `tie_failed`/`no_vote` путь, игра корректно продолжается.
- Победитель голосования (не финал): корректное исключение и возврат в игру/финал по правилам.
- Финальная ветка (`final_civilians_lose` / победа мирных): корректный game over splash.
- Reconnect/F5 во время collecting/result: UI восстанавливается через snapshot без рассинхрона.
